The other day I was handed a task to build a component that is mobile first but should work on IE8 and up. At first it sounded great, I was going to use a few small CSS features that are widely supported on mobile but not supported on IE8, let alone IE11. Most of it was a learning experience, by going through and getting 90% of it done and working in mobile chrome on android, chrome on desktop and safari on iOS. It wasn't until I pulled it up on IE8 in Browserstack, that I started to learn how creul the kind of CSS you are limited to on IE8.

IE8 is a weird browser, its one that supports a lot of things that seem quite new, or at least haven't been over used by the CSS comunity at large (like em), but it also doesn't support a lot of other things as well. I had a specific use case where I wanted to throw in a bit of progressive enhanced layout sugar into the CSS. I was tasked with making the component work with two parts that when joined together combined make up a 100% width component. I started with using calc, telling the first part of the component to be width: calc(100% - 6em); and the second part of the component with a width of 6em. This works perfectly for most browsers, until you reach IE8.

The first step I thought of was to declare the widths a second time, move the calc expression to be the second width declaration and on the first one I decided to go with a simple 70/30 percentage width split, this way the component works in IE8 and other browsers but doesn't need to look perfect in those browsers. The issue was however that browsers that supported em units but not calc would have the first part of the component 70% width and the second part 6em because they would parse the second width declaration on the second part of the component.

It was here that I found an interesting little CSS hack that I am calling a feature lock, more explicitly a one line feature lock. This kind of hack can be done in a few different ways that I know of so far, however it only works with CSS functions.
